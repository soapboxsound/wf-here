PART A — FIRESTORE SECURITY RULES (manual — do in Firebase console)

Go to your Firebase console. Click Firestore Database then the Rules tab. Replace the current test mode rules with these production rules. Click Publish after pasting.

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    match /listings/{listing} {
      allow read: if resource.data.status == 'approved';
      allow write: if false;
    }

    match /submissions/{submission} {
      allow create: if true;
      allow read, update, delete: if false;
    }

    match /savedSpots/{savedSpot} {
      allow read, write: if request.auth != null && request.auth.uid == resource.data.userId;
      allow create: if request.auth != null && request.auth.uid == request.resource.data.userId;
    }

    match /featuredLeads/{lead} {
      allow create: if true;
      allow read, update, delete: if false;
    }

    match /removalRequests/{request} {
      allow create: if true;
      allow read, update, delete: if false;
    }
  }
}

These rules mean: anyone can read approved listings. Nobody can write listings directly from the client. Anyone can submit a spot or a lead form. Only the owner can read and modify their own saved spots. Admin operations on submissions happen through the Firebase console directly, not through client code.

IMPORTANT — after publishing these rules the admin queue page will stop working because the rules block reading submissions from the client. To fix this tell Cursor: "Update /admin/queue/index.html to use the Firebase Admin SDK via a Vercel serverless function instead of reading Firestore directly from the browser. Create a Vercel API route at /api/admin/submissions.js that returns pending submissions, protected by a hardcoded admin password stored as a Vercel environment variable called ADMIN_PASSWORD."

Create /sw.js (service worker) in the project root. The service worker caches the core assets so the site loads offline or on slow connections. Cache these files on install: /, /explore/new-york, /css/tokens.css, /css/global.css, /css/components.css, /js/main.js, /js/firebase.js, /js/auth.js, /js/db.js, /js/components.js, /js/pins.js, /js/nav-auth.js, /assets/logo-primary.svg, /assets/logo-secondary.svg, /assets/favicon.png.

Use a cache-first strategy for assets and a network-first strategy for HTML pages. Cache name: "wf-here-v1".

Register the service worker in /js/main.js: if "serviceWorker" in navigator then register /sw.js on window load.

ADMIN PAGE PROTECTION

Add basic password protection to /admin/queue/index.html and /admin/listings/index.html.

At the very top of each admin page body before any content add a div with id "adminGate": a centered div, background var(--color-surface), padding 3rem, max-width 360px, margin 4rem auto. An h2 "Admin access" — font-size var(--text-xl), font-weight var(--weight-light), margin-bottom 1rem. A password input with id "adminPasswordInput" — same input styles. A button "Enter" with id "adminEnterBtn" — black button styles.

Set the correct password as a constant in the page script: const ADMIN_PASSWORD = "CHANGE_THIS_TO_YOUR_PASSWORD". Replace this with a real password before launch.

On page load check sessionStorage for "adminAuthenticated" equals "true". If found hide the admin gate and show the page content. If not found hide all page content and show only the admin gate.

When enter is clicked compare the input value to ADMIN_PASSWORD. If correct store "adminAuthenticated" in sessionStorage and hide the gate and show content. If wrong show an error "Incorrect password." in red text below the input.

Note: this is basic client-side protection — good enough for a personal admin panel. Do not use this for anything sensitive. The Firestore security rules from Part A provide the real protection.
